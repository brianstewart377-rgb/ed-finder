# Stage 19 — Data Warehouse Utopia Roadmap

## Goal

Stage 19 turns the ED-Finder data warehouse from passive scaffolding into the automated data engine for the project.

The target state is:

- external data imports run automatically;
- every source import is tracked by a source-run ledger;
- raw/source evidence is retained or reproducibly hashed;
- staging tables hold source facts without immediately changing canonical data;
- warehouse tables make source facts queryable and comparable;
- reconciliation produces reviewable candidates, not surprise writes;
- canonical updates remain controlled, reviewable, and auditable;
- failures produce logs/artifacts instead of silent corruption;
- admin/operator views show freshness, failures, coverage, and pending review work.

## Current assessment

The current warehouse is not operational enough yet.

There is useful scaffolding:

- enrichment code;
- staging concepts;
- source evidence patterns;
- operator artifacts;
- some provenance columns and evidence models;
- successful proof from Stage 18J/P18 that staged EDSM station evidence can drive controlled canonical improvements.

But the warehouse is not yet doing the full job:

- imports are not automatic;
- source-run tracking is not complete enough;
- raw/source retention is not yet systematic;
- freshness is not visible enough;
- failures are not surfaced clearly enough;
- canonical reconciliation still depends heavily on manual operator blocks;
- admin visibility is limited.

Stage 19 should therefore be treated as a build-out stage, not a small polish stage.

## Name

Stage 19 working name:

Data Warehouse Utopia

This means the system should aim for the ideal long-term shape, even if implementation is phased.

## Core principles

### 1. Source evidence first

External data should enter ED-Finder as source evidence before it affects canonical data.

Source data should be stored with:

- source name;
- source run key;
- source file key or URL key;
- source record hash;
- first seen timestamp;
- last seen timestamp;
- source updated timestamp when available;
- import timestamp;
- import status;
- row counts;
- error details if applicable.

### 2. Canonical data must remain protected

Automatic imports should not directly overwrite canonical truth.

Instead, imports should produce:

- staging rows;
- warehouse facts;
- reconciliation candidates;
- review packets;
- approval allowlists;
- controlled write-reviewed executions.

### 3. Every import must be auditable

Every import run should answer:

- what source was imported?
- when did it start and finish?
- what file/API/page was used?
- what hash identifies the source input?
- how many rows were read?
- how many rows were accepted?
- how many rows were rejected?
- what changed since last run?
- what failed?
- what artifacts were generated?

### 4. Automation must be boring

Auto-import should be safe, bounded, and boring.

It should:

- refuse unsafe runs;
- avoid overlapping imports;
- create artifacts every time;
- log clearly;
- not print secrets;
- not change canonical rows directly;
- be easy to disable;
- be visible from operator/admin status.

### 5. Reviewable reconciliation beats clever magic

The warehouse should make ED-Finder smarter, but not reckless.

When source evidence suggests canonical changes, the system should produce reviewable outputs first.

Examples:

- station external identity;
- station type;
- body slots;
- facility / service availability;
- economy;
- market / shipyard / outfitting;
- stale data warnings;
- source conflicts.

## Import source categories

Data Warehouse Utopia must classify every input stream by trust and purpose.

Source categories:

| Category | Meaning |
|---|---|
| Source of truth | Treated as authoritative for rules or definitions. |
| Source of evidence | Useful factual evidence, but still staged and reviewed before canonical updates. |
| Source of inspiration | Useful for UI/UX/tooling ideas, not canonical truth. |
| Manual/operator source | Human-curated artifacts, review packets, allowlists, and closeouts. |
| Derived source | Data generated by ED-Finder from other evidence. |

## Import source discovery and trust matrix

Stage 19 must include a dedicated import source discovery track.

The source matrix should record:

- source name;
- source category;
- access method;
- refresh cadence;
- rate limits;
- trust level;
- coverage;
- available fields;
- canonical impact;
- conflict risk;
- staging table required;
- warehouse table required;
- auto-import readiness;
- review requirement before canonical use;
- failure mode;
- operator visibility requirement.

## Source-specific initial notes

### EDSM

Recommended first automated stream.

Reasons:

- already present in staging;
- already proved useful for station external identity;
- already provided station type evidence;
- supports the recent Dodec/Coriolis controlled write flow.

Initial automation target:

- EDSM station/system enrichment into staging/warehouse;
- source-run ledger;
- artifact per import;
- no direct canonical writes.

### Spansh

Potential second major stream.

Likely value:

- large-scale system/body coverage;
- coordinates;
- body data;
- useful for search and route/system discovery.

Risks:

- large datasets;
- memory/runtime pressure;
- stale snapshots;
- schema drift;
- may require chunked import.

### Inara

Potential high-value enrichment stream.

Likely value:

- station services;
- economies;
- market/faction signals;
- possible useful metadata where access allows.

Risks:

- access method and rate limits must be reviewed;
- trust and freshness must be explicit;
- should not directly overwrite canonical data.

### DaftMav / facility template data

Likely important for planning logic.

Use as:

- build template evidence;
- facility cost/effect reference;
- simulation/planner source.

Must be:

- versioned;
- hashed;
- imported as structured source rows;
- kept separate from game-live station/system data.

### Mega Guide

Treat as rules/reference source.

Use for:

- mechanics;
- constraints;
- interpretation;
- validation rules.

Do not treat as automatically-refreshing factual source unless manually converted into structured rule tables.

### RavenColonial

Treat as inspiration only.

Use for:

- UI ideas;
- workflow ideas;
- possible feature comparison.

Do not treat as source of canonical Elite Dangerous data.

## Stage 19 track overview

### Stage 19A — Warehouse readiness audit

Purpose:

- inventory existing warehouse/staging/importer code;
- inventory DB tables and provenance columns;
- inspect source-run support;
- inspect existing import entrypoints;
- identify what is real, stale, or unused;
- produce a factual artifact.

Boundary:

- read-only;
- no imports;
- no DB writes;
- no migrations;
- no canonical apply.

### Stage 19B — Target warehouse architecture

Purpose:

Define the intended warehouse architecture clearly before implementation expands.

Must specify:

- raw/source layer;
- staging layer;
- warehouse layer;
- canonical layer;
- reconciliation layer;
- artifact layer;
- admin/operator visibility layer.

Deliverables:

- architecture doc;
- source-run ledger contract;
- table naming rules;
- artifact naming rules;
- import safety rules;
- failure-state rules.

### Stage 19C — Import source discovery, priority, and trust matrix

Purpose:

Build the full source stream inventory before automating only one source.

Deliverables:

- import source matrix;
- trust classification;
- access method;
- refresh cadence;
- staging needs;
- warehouse needs;
- canonical impact;
- risk score;
- implementation priority.

This is where previously considered streams must be recorded explicitly.

### Stage 19D — Source-run ledger

Purpose:

Create the durable import-run tracking foundation.

A source-run ledger should track:

- run key;
- source name;
- source type;
- source URL/path/API endpoint;
- source input hash;
- started at;
- finished at;
- status;
- row count read;
- row count staged;
- row count rejected;
- artifact path;
- artifact hash;
- error summary;
- importer version;
- git commit SHA;
- operator/scheduler context.

### Stage 19E — Idempotent importer wrapper

Purpose:

Create one safe wrapper pattern for imports.

Requirements:

- no duplicate source-run keys;
- no overlapping import for the same source;
- validate source hash;
- write staging first;
- produce artifact;
- fail safely;
- never apply canonical changes directly;
- support dry-run/report-only mode;
- support bounded write mode only when explicitly designed.

### Stage 19F — EDSM auto-import MVP

Purpose:

Make one real source automatic.

Likely scope:

- EDSM station/system enrichment;
- staging tables only;
- source-run ledger;
- artifact per import;
- freshness summary;
- failure summary.

Explicit non-scope:

- no canonical station updates;
- no station-type writes;
- no canonical apply;
- no broad reconciliation write.

### Stage 19G — Freshness and coverage artifacts

Purpose:

Generate regular warehouse status artifacts.

Should answer:

- when was each source last imported?
- when did each source last succeed?
- when did each source last fail?
- how many records were staged?
- how stale is each source?
- what changed since the previous run?
- what evidence is missing?
- what conflicts exist?

### Stage 19H — Auto-import scheduler

Purpose:

Run safe imports automatically.

Preferred approach:

- systemd timer on Hetzner;
- one source at a time;
- lockfile to prevent overlap;
- logs to a fixed location;
- artifacts to operator artifact directory;
- disabled/reviewed before production enablement.

Scheduler must not:

- print secrets;
- run canonical apply;
- update canonical rows directly;
- continue after source validation failure.

### Stage 19I — Admin/operator visibility

Purpose:

Expose warehouse health in the app/admin UI.

Useful panels:

- latest source imports;
- source freshness;
- failed imports;
- staged row counts;
- source-run ledger;
- pending reconciliation candidates;
- station-type candidate counts;
- unknown/refused/deferred source values;
- last write-reviewed execution;
- artifact links/hashes;
- current safety state.

### Stage 19J — Reconciliation candidate engine

Purpose:

Turn warehouse facts into reviewable candidate artifacts.

Candidate types:

- station external identity;
- station type;
- station services;
- market availability;
- landing pad;
- economy;
- system/body metadata;
- body-slot signals;
- facility/build-plan source evidence.

Each candidate should include:

- source evidence;
- current canonical value;
- proposed value;
- confidence;
- conflict reason;
- freshness;
- provenance;
- review questions.

### Stage 19K — Controlled canonical update lanes

Purpose:

Create controlled lanes for different update categories.

Examples:

- station type lane;
- external identity lane;
- station service lane;
- economy lane;
- body data lane;
- build-plan/facility lane.

Each lane should have:

- dry-run artifact;
- review packet;
- approval allowlist;
- bounded write-reviewed execution;
- post-write verification;
- docs closeout.

### Stage 19L — Tests and refusal policies

Purpose:

Make safety policies testable.

Carry forward Stage 18 policies:

| Source value | Policy |
|---|---|
| `Coriolis Starport` | May map to `Coriolis` with confirmed identity and review/approval. |
| `Dodec Starport` | May map to `Dodec` with confirmed identity and review/approval. |
| `Drake-Class Carrier` | Refuse/defer as transient/mobile carrier. |
| `Space Construction Depot` | Refuse/defer as transient construction object. |
| `NULL` | Refuse. |

### Stage 19M — Documentation/runbooks

Purpose:

Make operator use less error-prone.

Runbooks should include:

- where to run commands;
- explicit Hetzner-only labels;
- Codex/Grok “do not run” warnings;
- expected working directory;
- required env variables;
- secret-safe output;
- artifact output path;
- success criteria;
- stop conditions;
- rollback/disable procedure.

### Stage 19N — Full automation hardening

Purpose:

Move from semi-manual safe automation to robust unattended operation.

Hardening items:

- scheduler lock;
- timeout handling;
- retry policy;
- alerting;
- stale-source warnings;
- artifact retention policy;
- disk usage checks;
- source API failure handling;
- poison-data rejection;
- admin dashboard integration.

## Implementation priorities

### Priority 1 — make the warehouse real

- Source-run ledger.
- Idempotent import wrapper.
- EDSM staging import.
- Read-only status artifacts.
- Import source matrix.

### Priority 2 — automate safely

- systemd timer;
- lockfile;
- logs;
- freshness artifacts;
- failure artifacts.

### Priority 3 — make it visible

- API/admin endpoint;
- UI/operator status panel;
- artifact references.

### Priority 4 — expand sources

- Spansh;
- Inara;
- facility/build-plan sources;
- additional high-value community sources.

### Priority 5 — scale reconciliation

- candidate engines;
- review packets;
- approval allowlists;
- controlled write lanes.

## Non-goals for early Stage 19

Do not start with:

- broad canonical apply;
- automatic canonical writes;
- UI mega-redesign;
- source import without provenance;
- source import without artifacts;
- source import without refusal rules;
- huge repo-wide refactor;
- parallel automation for every source at once.

## Definition of done for Stage 19

Stage 19 can be considered successful when:

- at least one high-value external source imports automatically;
- every import has a source-run record;
- every import creates an artifact;
- staging/warehouse data is queryable;
- canonical data is not changed by import alone;
- freshness and failure status are visible;
- operator can inspect current warehouse state easily;
- reconciliation candidates can be generated from warehouse facts;
- controlled write lanes remain reviewable and auditable.
